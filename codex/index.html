Blok 2 ukázal, co AI zvládne v prohlížeči. Vždy ale přijde ten samý strop:

Kdo, co, jak — každá nová konverzace začíná vysvětlováním od nuly.

Zapomíná

Soubory, e-maily, poznámky — vše je nutné nakopírovat ručně.

Chat = výborný poradce. Špatný kolega.

Co je Second Brain

Externalizovaná, strukturovaná, plain-text paměť pracovní reality — projekty, lidé, rozhodnutí, znalosti — kterou čte a zapisuje člověk i AI agent.

Není to poznámkovač
Je to pracovní prostor, do kterého se agent zaučí.

Není to další appka
Je to složka souborů na disku — čitelná kýmkoli i čímkoli.

„Myšlení můžeš outsourcovat.
Porozumění ne." — Andrej Karpathy

Second Brain je vrstva, kde se drží porozumění — struktura, rozhodnutí, kontext.
AI pak dělá myšlení a práci.

Markdown — základ Second Brainu

Takto vypadá zdrojový text:

# Projekt Alpha

**Cíl:** zvýšit konverzi o 15 %

## Akční kroky
- [ ] Analýza funnelu
- [ ] A/B test přihlášení

> Rozhodnutí 15. 5.: jdeme cestou B

Proč plain text + MD?

  • Čitelný člověkem i AI — bez konverze
  • Git diff ukáže přesně co se změnilo
  • Otevře ho jakýkoli editor — Obsidian, VS Code, Notepad
  • Žádný import/export, žádný vendor lock-in
  • Za 30 let bude stále čitelný

Psací stroj Second Brainu — jednoduchý jako poznámkový blok, bohatý jako wiki.

 Pro pokročilé (šipka dolů): frontmatter · Mermaid · MARP

Frontmatter — metadata v každém souboru

Každý MD soubor může začínat YAML hlavičkou:

---
type: meeting
created: 2026-06-02
participants: [Jana, Petr]
tags: [projekt-alfa, q2-planning]
status: done
abstract: >
  Q2 roadmapa. 3 závazky,
  termín 30. 6.
---

# Obsah schůzky...

Co to umožní:

  • Agent filtruje soubory podle type, status, tags
  • Dotaz: „Najdi všechna rozhodnutí z května"
  • Git sleduje historii — je vidět, kdy se status změnil
  • Automatizace: pracovat jen se soubory, kde status: open

Frontmatter = strukturovaná data uvnitř textu.
Jeden soubor, dvě vrstvy — metadata + obsah.

 Dál dolů: rozšíření Markdownu — Mermaid · MARP

Mermaid  ·  MARP — co MD ještě umí

Mermaid — diagramy jako text

flowchart TD
  A[Nahrávka] --> B[Přepis]
  B --> C{Souhlas?}
  C -->|ok| D[(Uložit)]
  C -->|uprav| B

Právě tento diagram v tomto decku — psaný jako plain text, renderovaný jako SVG.

Agent ho čte i píše — může generovat vizualizaci přímo z dat.

Typy: flowchart, org chart, ER diagram, Gantt, timeline, sekvence.

MARP — prezentace z Markdownu

---
marp: true
theme: default
---

# Slide 1

Obsah prvního slidu.

---

# Slide 2

Každý `---` = nový slide.

Celá prezentace = jeden .md soubor. Verzovaný v gitu. Agent ho přečte i napíše.

Stejný princip jako tento deck — ale čistý text bez HTML. Markdown → slidy jedním příkazem.

Co do brain nepatří

Nikdy do MD souborů:

  • Hesla, API klíče, přístupové tokeny
  • Konfigurační soubory .env
  • Osobní údaje klientů (GDPR)
  • Čísla karet, bankovní přihlašovací údaje
  • Data konkurenčně citlivá bez souhlasu

Proč na to dávat pozor:

Brain je verzovaný v gitu. Může být synchronizovaný do cloudu — iCloud, OneDrive, GitHub. Codex i každý agent ho čte.

Citlivá data patří do správce hesel (1Password, Bitwarden) nebo šifrovaného trezoru — ne do plain textu.

Test: Kdyby se soubor náhodně dostal ke kolegovi — byl by to problém? Pokud ano — nepatří do brainu.

Pět principů

Proč to stavět jako složku souborů — a ne uvnitř něčí appky.

① Platformní nezávislost — vlastni substrát

Nejdůležitější princip ze všech pěti.

Soubory patří uživateli. Navždy.
Prostý text. Žádný proprietární export, žádný lock-in.

AI je vyměnitelný engine, mozek ne.
Zvedne OpenAI cenu? Stačí namířit ty samé soubory na jiného agenta. Nulová migrace.

Investice se přenáší.
AGENTS.md čte dnes Codex i Claude — obě laboratoře k ní dospěly nezávisle.

Dnes vybíráme Codex — v češtině, GDPR/DPA, data se netrénují pro trénink modelů.

Ale brain postavíme tak, abychom na Codexu nikdy nebyli závislí.

To je rozdíl mezi nástrojem a pastí.

② Jeden zdroj pravdy

Syrový vstup → rozpracované → hotový artefakt. Nikdy míchat.

_inputs → _wip → _outputs

Když číslo nesedí, jde se ke zdroji — ne k další kopii kopie.

③ Kontext je palivo

Polovina výkonu modelu je „dostat správný kontext ve správný čas".

Proto AGENTS.md — krátká příručka, kterou agent přečte na startu každého sezení.

Analogie: nový asistent, první den — dostane příručku kdo je kdo a jak se co dělá. AGENTS.md je ta příručka. Rozdíl mezi agentem, kterého je nutné brífovat znovu a znovu, a agentem, který už ví.

④ Paměť přežívá sezení     ⑤ Skládání

④ Paměť přežívá sezení

Agent nemá trvalou paměť — externalizuje se do souborů.

Fakta o lidech a projektech, poučení, závazky. Příště si je agent sám přečte.

Znalost se neresetuje s každým novým chatem.

⑤ Skládání

Co funguje, ulož jako workflow / skill. Otravná opakovaná práce → jednou popsaný postup, který agent spustí na povel.

Z chyb se stávají položky v kontrolním seznamu. Systém se v čase zlepšuje sám.

Každé sezení zrychluje ta příští.

Jak to vypadá — struktura složek

flowchart TD ROOT(["📁 Second Brain"]) --> AG["📄 AGENTS.md"] ROOT --> INFRA["🤖 AI Infrastruktura"] ROOT --> OBSAH["🗂️ Pracovní obsah"] INFRA --> MEM["0_AI_Memory"] INFRA --> SK["0_AI_Skills"] OBSAH --> PROJ["1_Projekty"] OBSAH --> JOUR["5_Žurnál"] OBSAH --> KNOW["6_Znalosti"] PROJ --> PX["Projekt X"] PX --> IN["📥 _inputs"] PX --> WP["⚙️ _wip"] PX --> OU["📤 _outputs"]

AI infrastruktura žije vedle obsahu — jako soubory, ne v cizí appce. Vzor _inputs → _wip → _outputs se opakuje napříč projekty.

Git — záloha i paměť systému

Git sleduje každou změnu v textových souborech. S Markdown brainem se stane neviditelnou záchrannou sítí.

  • Historie — záznam každé změny, kdo a kdy
  • Záloha — push na GitHub = bezpečná kopie v cloudu
  • Diff — AI vidí přesně co se od minula změnilo
  • Rollback — omyl? Návrat k libovolné verzi

Tři příkazy, které stačí:

git add .
git commit -m "zápis z porady"
git push

Nebo GUI bez terminálu: GitHub Desktop (Mac + Windows) — jen klikání, žádný příkazový řádek.

Brain bez gitu = deník bez zálohy.
Brain s gitem = deník s časovým strojem.

Speciální soubory — jazyk agenta

Každý opakující se kontext, charakter nebo workflow má svůj pojmenovaný MD soubor. Spolu tvoří systém, který drží kontext, charakter i postupy.

design.md

Dokumentuje proč, ne jen co.

SKILL.md

Skill není kód. Je to strukturovaný markdown s instrukcemi.

## Kdy použít
Když přijde Plaud transkript.

## Kroky
1. Oprav překlepy
2. Ulož raw přepis
3. Recall kontextu
...

## Anti-patterns
Nevolat get_transcript zbytečně.

Jeden soubor navíc → nová schopnost agenta.

Skilly — naučená schopnost, navždy

Skill je opakovatelný postup, který agenta naučí zvládat určitý druh úlohy. Popsaný jednou — použitelný pořád.

Jako zaškolení kolegy: postup se sepíše jednou, pak ho zvládne kdokoli — pokaždé stejně.

Spouští se na povel — jménem nebo příkazem (/report, /zpracuj-hovor). Agent často sám pozná, kdy skill nasadit.

Je to složka: SKILL.md (kdy použít · kroky · anti-patterns) + volitelně skripty a šablony.

Příklady skillů:

📥 Inbox sweep — projít maily, nadraftovat odpovědi

📊 Týdenní report — stáhnout metriky, sestavit přehled

🔍 Rešeršní brief — z odkazů sestavit ozdrojovaný podklad

🎙️ Zpracovat nahrávku — přepis → úkoly → zápis (za chvíli naživo)

Knihovna skillů roste a sdílí se v týmu. Každý nový skill = trvalá schopnost navíc.

Skládání v praxi

Konkrétní příklad principu ⑤ — co přinese, když se opakující práce jednou popíše jako workflow.

Plaud — od nahrávky k trvalé paměti

Plaud je malý hlasový diktafon.

Po každé poradě 30 minut zvuku — a kdysi i 30 minut otravného přepisování, třídění do poznámek, hledání závazků.

Dnes je to jeden povel.

Agent dostane nahrávku, projde workflow, uloží vše na správné místo — a čeká na souhlas před každým zápisem.

Co workflow udělá:

  • Opraví přepis, doptá se na nejasnosti
  • Uloží syrový transkript
  • Vytáhne kontext z minulých sezeních
  • Destiluje rozhodnutí, závazky, fakta
  • Uloží na 3 místa: paměť · úkoly · zápis z porady
  • Ověří, že vše fyzicky existuje

Workflow Plaud skillu

flowchart TD A["🎙️ Nahrávka"] --> B["Přepis a oprava"] B --> C[("💾 Raw přepis")] C --> D["🔎 Recall kontextu"] D --> E["Destilace"] E --> F{"Souhlas?"} F -->|"uprav"| E F -->|"ok"| G[("🧠 Supermemory")] F -->|"ok"| H[("✅ Linear")] F -->|"ok"| I[("📝 Zápis z porady")] G --> J["Finální audit"] H --> J I --> J J --> K(["✓ Dohledatelné"])

Uzel „Souhlas?" — agent nic neuloží bez lidského schválení. Audit ověří, že soubory fyzicky vznikly.

Čím víc se brain používá, tím víc vrací

Second Brain funguje jako složené úročení — každý zápis zvětšuje základ, ze kterého se příště čerpá.

Měsíc 1

Prázdná složka.

Pár souborů. Agent se brífuje ručně při každém sezení.

Rok 1

Kontext se hromadí.

Agent zná projekty, lidi, rozhodnutí. Čím dál méně opakování.

Rok 2+

Systém zná kontext.

Agent navrhne řešení dřív, než je celé popsané. Brain ví víc než lidská paměť.

Agent má víc kontextu → lepší odpovědi → lepší rozhodnutí → víc zápisů → agent má víc kontextu. — smyčka složeného učení

Každý zápisek je malá investice. Neurčuje výsledek sám — ale zvyšuje základ, z nějž příští rozhodnutí vychází. Na rozdíl od chat historií — tahle paměť nevyprší.
